# BidPool 标讯智能平台技术文档

## 1. 系统架构

BidPool 采用三层微服务架构，分为 Go 后端、Python 智能体服务和 Vue3 前端，三者通过 HTTP API 通信。

```
┌──────────────────────────────────────────┐
│            Vue3 Frontend                 │
│     (bidpool-web, port 8088)             │
└────────────────┬─────────────────────────┘
                 │ REST API (JSON)
                 ▼
┌──────────────────────────────────────────┐
│         Go Backend Service               │
│  Gin Router → Handlers → Scheduler      │
│                        → SQLite (GORM)   │
│  port 8088                               │
└────────────────┬─────────────────────────┘
                 │ HTTP (Python Service URL)
                 ▼
┌──────────────────────────────────────────┐
│       Python Agent Service               │
│  FastAPI Router                          │
│  ├─ CrawlerAgent (subprocess isolation)  │
│  ├─ FilterAgent                          │
│  ├─ DispatchAgent → DingTalk Webhook     │
│  └─ OrchestratorAgent (LLM + memory)     │
│  port 8000 (internal)                    │
└──────────────────────────────────────────┘
```

**关键设计原则**：
- Go 后端负责业务逻辑编排、数据持久化和定时调度
- Python 智能体负责需要 AI/浏览器的复杂操作（爬虫、LLM 对话）
- 前端通过 Go API 统一入口，不直接调用 Python 服务
- 所有跨服务调用通过 HTTP JSON 交互

## 2. Go 后端 (bidpool-go)

### 2.1 启动入口

`cmd/server/main.go` 是应用入口，按顺序完成：加载配置 → 初始化数据库 → 启动调度器 → 启动 API 服务。

```go
func main() {
    configPath := os.Getenv("CONFIG_PATH")
    if configPath == "" {
        configPath = "configs/config.yaml"
    }

    cfg, err := config.LoadConfig(configPath)
    db, err := store.InitDB(cfg.Database.Path)

    sched := scheduler.NewScheduler()
    sched.Start()

    router := api.SetupRouter(db, sched, cfg)
    router.Run(":" + cfg.Server.Port)
}
```

### 2.2 配置管理

使用 YAML 文件配置，`config.LoadConfig()` 读取并设置默认值。配置结构体嵌套定义了所有模块的配置项。

```go
type Config struct {
    Server    ServerConfig    `yaml:"server"`    // 端口、主机
    Database  DatabaseConfig  `yaml:"database"`  // SQLite 路径
    Agent     AgentConfig     `yaml:"agent"`     // Python 服务地址
    Auth      AuthConfig      `yaml:"auth"`      // Basic Auth 凭证
    Platforms PlatformsConfig `yaml:"platforms"` // 爬虫平台账号

```

`LoadConfig()` 会为未设置的字段提供默认值（如 `Server.Port` 默认为 `8080`）。

### 2.3 路由与 API 设计

`api.SetupRouter()` 使用 Gin 框架注册路由，分为三部分：

- **静态文件服务**：`/assets`、`/favicon.ico` — 不需要认证
- **API 路由组** `/api/v1`：条件性地启用 Basic Auth 中间件
- **SPA 回退**：`NoRoute` 将所有非 API 请求指向 `index.html`

关键代码：

```go
func SetupRouter(db *gorm.DB, sched *scheduler.Scheduler, cfg *config.Config) *gin.Engine {
    r := gin.Default()
    r.Static("/assets", "./dist/assets")

    // 创建 handlers
    taskHandler := handler.NewTaskHandler(db, sched, cfg)
    bidHandler := handler.NewBidHandler(db)

    // 恢复定时任务
    taskHandler.RestoreScheduledTasks()

    // API 路由组
    v1 := r.Group("/api/v1")
    if cfg.Auth.Enabled {
        v1.Use(middleware.BasicAuth(cfg.Auth.Username, cfg.Auth.Password))
    }
    {
        v1.POST("/tasks", taskHandler.CreateTask)
        v1.GET("/tasks", taskHandler.ListTasks)
        v1.POST("/tasks/:id/run", taskHandler.RunTask)
        v1.GET("/bids", bidHandler.ListBids)
        v1.POST("/chat", chatHandler.HandleChat)
        v1.GET("/status", statusHandler.GetStatus)
    }

    // SPA 回退
    r.NoRoute(func(c *gin.Context) {
        c.File("./dist/index.html")
    })
    return r
}
```

### 2.4 定时任务调度器

使用 `robfig/cron/v3` 库，`Scheduler` 结构体封装了任务的增删改查。关键设计：

- 支持 5 字段和 6 字段 Cron 表达式（自动补全秒字段）
- 线程安全的任务映射（`sync.RWMutex`）
- 服务启动时自动恢复数据库中的已启用任务

```go
type Scheduler struct {
    cron *cron.Cron
    jobs map[string]cron.EntryID
    mu   sync.RWMutex
}

func (s *Scheduler) AddJob(id string, spec string, cmd func()) error {
    s.mu.Lock()
    defer s.mu.Unlock()

    // 如果任务已存在，先移除再添加
    if _, exists := s.jobs[id]; exists {
        s.cron.Remove(s.jobs[id])
    }

    // 自动补全秒字段（5字段→6字段）
    fields := strings.Fields(spec)
    if len(fields) == 5 {
        spec = "0 " + spec
    }

    entryID, err := s.cron.AddFunc(spec, cmd)
    s.jobs[id] = entryID
    return err
}
```

任务恢复逻辑：从数据库查询所有 `enabled=true` 且 `cron_expr` 非空的任务，逐个注册到调度器。

```go
func (h *TaskHandler) RestoreScheduledTasks() {
    var tasks []model.TaskConfig
    h.db.Where("enabled = ? AND cron_expr != ''", true).Find(&tasks)
    for _, task := range tasks {
        h.sched.AddJob(task.ID, task.CronExpr, func() {
            h.executeTaskWithTracking(task.ID)
        })
    }
}
```

### 2.6 任务执行流程

任务执行分为**爬取任务**和**发送任务**两种类型，统一入口是 `executeTaskWithTracking()`：

```go
func (h *TaskHandler) executeTaskWithTracking(taskID string) {
    h.exec.StartExecution(taskID)
    h.exec.AddLog(taskID, "info", "开始执行任务")

    // 创建数据库执行日志
    log := model.ExecutionLog{
        ID:     uuid.New().String(),
        TaskID: taskID,
        Status: "running",
    }
    h.db.Create(&log)

    if taskType == "send" {
        h.executeSendTask(taskID, task, &log)
    } else {
        h.executeCrawlTask(taskID, task, &log)
    }
}
```

**爬取任务流程**：

1. 解析参数：日期计算（动态或固定）、地区编码
2. 调用 Python Agent：`POST /agent/crawler/execute`
3. 解析返回结果，提取标讯列表
4. 逐条保存到 SQLite（ID 唯一检查，已存在则跳过）
5. 更新执行日志和状态追踪

**发送任务流程**：

1. 从数据库查询待发送标讯（支持地区/行业/日期筛选）
2. 调用 Python Agent：`POST /agent/dispatch/send`
3. 发送成功后更新标讯状态为 `sent`

两个流程都包含详细的**步骤追踪**，通过 `execution.Manager` 记录每一步的状态（`running/success/failed`）和日志，前端可实时查看。

### 2.7 执行状态追踪

`execution.Manager` 提供内存中的执行状态管理，记录每个任务的步骤进度、日志和最终状态：

```go
// 关键方法
StartExecution(taskID string)     // 开始追踪
UpdateStep(taskID, step, status, msg) // 更新步骤
AddLog(taskID, level, message)   // 添加日志
FinishExecution(taskID, status, count, msg) // 完成
```

Go Handler 端调用 Python Agent 时，会依次更新步骤状态，使得前端可以实时展示执行进度（如"正在登录..."→"登录成功"→"搜索完成"→"数据保存完成"）。

## 3. Python 智能体服务 (bidpool-python)

### 3.1 Agent 基类

所有 Agent 继承自 `BaseAgent`，定义了统一的执行流程：validate → pre_process → execute → post_process。

```python
class BaseAgent(ABC):
    name: str = "base_agent"

    @abstractmethod
    def execute(self, task_data: Dict[str, Any]) -> AgentResult:
        pass

    def run(self, task_data: Dict[str, Any]) -> AgentResult:
        if not self.validate_input(task_data):
            return AgentResult(success=False, message="输入数据验证失败")
        processed_data = self.pre_process(task_data)
        result = self.execute(processed_data)
        return self.post_process(result)
```

`AgentResult` 使用 dataclass 定义，包含 `success`、`message`、`data`、`error` 四个字段，统一了所有 Agent 的返回格式。

### 3.2 爬虫智能体 (CrawlerAgent)

CrawlerAgent 通过 `CrawlerRegistry` 管理多个平台的爬虫实现。目前已注册千里马爬虫。

```python
class CrawlerAgent(BaseAgent):
    name = "crawler_agent"

    def __init__(self, config=None):
        super().__init__(config)
        self.registry = CrawlerRegistry()
        self.registry.register("qianlima", QianliMaCrawler)

    def execute(self, task_data):
        platform = task_data.get("platform", "qianlima")
        crawler = self.registry.get_crawler(platform)

        # 登录
        credentials = self._get_credentials(platform, task_data.get("credentials"))
        if not crawler.login(credentials):
            return AgentResult(success=False, message="登录失败")

        # 搜索
        bids = crawler.search({
            "region_codes": task_data.get("region_codes", []),
            "start_date": task_data.get("start_date"),
            "end_date": task_data.get("end_date")
        })
        return AgentResult(success=True, message=f"成功爬取 {len(bids)} 条标讯",
                           data={"bids": bids, "count": len(bids)})
```

凭证获取有两级回退：优先使用请求中传递的凭证，否则从配置文件读取。

### 3.3 Playwright 子进程隔离（关键难点与解决方案）

**问题背景**：Playwright 的同步 API 在 uvicorn（FastAPI 的异步服务器）环境中无法直接使用。因为 uvicorn 的 `asyncio` 事件循环被 `fork()` 继承，Playwright 检测到当前进程已有事件循环时会抛出异常：

```
It looks like you are using Playwright Sync API inside the asyncio loop
```

**解决方案**：通过 `asyncio.create_subprocess_exec()` 启动完全隔离的子进程来运行 Playwright，使用临时文件传递结果。

```python
async def _run_crawler_subprocess(task_data):
    runner_path = "/app/crawlers/runner.py"
    result_file = os.path.join(
        tempfile.gettempdir(),
        f"crawl_result_{os.getpid()}_{id({})}.json"
    )
    task_data["_result_file"] = result_file
    task_json = json.dumps(task_data, ensure_ascii=False)

    proc = await asyncio.create_subprocess_exec(
        sys.executable, runner_path,
        stdin=asyncio.subprocess.PIPE,
        stdout=asyncio.subprocess.PIPE,
        stderr=asyncio.subprocess.PIPE
    )

    stdout, stderr = await asyncio.wait_for(
        proc.communicate(input=task_json.encode("utf-8")),
        timeout=180
    )

    # 从临时文件读取结果
    if os.path.exists(result_file):
        with open(result_file, "r", encoding="utf-8") as f:
            output = f.read()
        os.remove(result_file)

    result_data = json.loads(output)
    return AgentResult(
        success=result_data.get("success", False),
        message=result_data.get("message", ""),
        data=result_data.get("data")
    )
```

子进程入口 `runner.py` 独立运行，通过 `stdin` 接收任务数据，将结果写入临时文件，只将文件路径打印到 `stdout`：

```python
# crawlers/runner.py
def run():
    task_json = sys.stdin.read()
    task_data = json.loads(task_json)

    result_file = task_data.pop("_result_file", None)

    # 在完全独立的环境中执行爬虫
    from agents.crawler_agent import CrawlerAgent
    from dataclasses import asdict

    agent = CrawlerAgent()
    result = agent.run(task_data)

    # 转换数据格式（dataclass → dict）
    data = result.data
    if data and "bids" in data:
        data["bids"] = [asdict(b) for b in data["bids"]]

    output = json.dumps({
        "success": result.success,
        "message": result.message,
        "data": data,
        "error": result.error
    }, ensure_ascii=False, default=str)

    with open(result_file, "w", encoding="utf-8") as f:
        f.write(output)

    # 仅输出文件路径，避免与库的 print() 输出混合
    print(result_file, flush=True)
```

**使用临时文件的原因**：ddddocr 等第三方库会直接向 `stdout` 输出文本（如"欢迎使用 ddddocr"），这会导致 JSON 解析失败。通过临时文件传递结果，`stdout` 只输出一行文件路径，完全避免混合。

### 3.4 编排智能体 (OrchestratorAgent)

OrchestratorAgent 是系统的"大脑"，负责：

1. **多轮对话**：维护会话记忆，支持上下文理解
2. **意图识别**：解析用户自然语言，映射到具体操作
3. **流程编排**：协调 Crawler/Filter/Dispatch Agent 的执行
4. **状态管理**：跟踪多步操作流程（如发送标讯的群选择→筛选→确认）

**多轮对话与记忆机制**：

```python
class OrchestratorAgent(BaseAgent):
    def __init__(self, config=None):
        self.llm_client = LLMClient()
        self.memory = get_memory()  # ConversationMemory 实例
        self.crawler_agent = CrawlerAgent(config)
        self.dispatch_agent = DispatchAgent(config)

    def process_chat(self, message, session_id="default"):
        # 记录用户消息
        self.memory.add_message(session_id, "user", message)

        # 检查当前对话阶段
        stage = self.memory.get_stage(session_id)
        pending_data = self.memory.get_pending_data(session_id)

        if stage == "waiting_group":
            return self._handle_group_selection(message, session_id)
        elif stage == "waiting_bids_filter":
            return self._handle_bids_filter(message, session_id)
        elif stage == "waiting_confirm":
            return self._handle_confirm(message, session_id)

        # 新请求 — 解析意图
        intent = self._parse_intent_with_context(message, session_id)
        # ... 根据 intent.type 分发到不同处理方法
```

**意图识别**有两种策略：优先使用 LLM（结合对话历史），降级使用关键词匹配：

```python
def _parse_intent_with_context(self, message: str, session_id: str) -> Dict:
    history = self.memory.get_history(session_id)
    if len(history) > 0:
        # 让 LLM 结合上下文解析意图
        context_str = "\n".join([
            f"{h['role']}: {h['content'][:100]}"

        prompt = f"""结合对话历史，分析用户最新消息的意图。
对话历史:\n{context_str}\n用户最新消息: {message}
返回JSON: {{"type": "意图类型", "params": {{参数}}}}"""
        # ...
```

**多步发送流程的状态机**：

```
用户说"发送标讯"
  → 阶段: waiting_group  → 询问选择哪个钉钉群
    → 阶段: waiting_bids_filter  → 询问筛选条件
      → 阶段: waiting_confirm  → 确认发送
        → 完成，清零状态
```

### 3.5 钉钉消息发送

`DingTalkSender` 封装钉钉机器人 Webhook 的所有消息类型：text、markdown、link、actionCard。

**关键挑战**：钉钉消息有 20000 字节限制。对于大量标讯，需要自动分批发送。

```python
def send_to_dingtalk(request: DispatchSendRequest):
    sender = DingTalkSender()
    message = sender.format_bid_message(request.bids)
    message_bytes = len(message.encode('utf-8'))

    if message_bytes > 19000:
        # 分批发送 - 每批10条
        batch_size = 10
        for i in range(0, len(request.bids), batch_size):
            batch_bids = request.bids[i:i+batch_size]
            batch_message = sender.format_bid_message(batch_bids)
            # 更新标题添加批次信息
            batch_message = batch_message.replace(
                "## 📋 标讯推送",
                f"## 📋 标讯推送（第 {batch_num}/{total_batches} 批）"
            )
            sender.send_markdown(webhook_url, title, batch_message)
    else:
        sender.send_markdown(webhook_url, "标讯推送", message)
```

**消息格式化**：`format_bid_message()` 将标讯列表转换为美观的 Markdown，每条包含预算、开标时间、招标单位、地区和详情链接。

### 3.6 FastAPI 路由

Python 服务使用 FastAPI，通过 `APIRouter` 组织路由：

| 路由 | 功能 |
|------|------|
| `POST /agent/crawler/execute` | 执行爬虫（子进程隔离） |
| `POST /agent/filter/execute` | 过滤标讯 |
| `POST /agent/dispatch/execute` | 分发任务 |
| `POST /agent/dispatch/send` | 直接发送到钉钉 |
| `POST /agent/pipeline/execute` | 完整流程（爬取→过滤→分发） |
| `GET /agent/status` | Agent 健康状态 |

`dispatch/send` 和 `pipeline/execute` 以外的路由被 Go 后端调用，不对外暴露。

## 5. 数据流完整示例

### 5.1 定时爬取流程

```
Cron 触发 (55 8 * * *)
  → Scheduler 调用 executeTaskWithTracking(taskID)
    → Go 创建 ExecutionLog (status=running)
    → Go 计算日期范围
    → Go HTTP POST → Python /agent/crawler/execute
      → Python 启动子进程 runner.py
        → 子进程: CrawlerAgent.run()
          → QianliMaCrawler.login()  → 登录千里马
          → QianliMaCrawler.search()  → 搜索标讯
        → 子进程: 结果写入临时文件 → 退出
      → Python 读取临时文件 → 返回给 Go
    → Go 解析标讯列表 → 逐条写入 SQLite（去重）
    → Go 更新 ExecutionLog (status=success, bids_count=N)
```

### 5.2 对话发送流程

```
用户在 Chat.vue 输入 "发送标讯"
  → Vue POST → Go /api/v1/chat
    → Go HTTP POST → Python /agent/pipeline/execute
      → OrchestratorAgent.process_chat()
        → 意图识别: send_bids
        → _start_send_flow(session_id)
          → 查询钉钉群配置
          → 设置阶段为 waiting_group
          → 返回群选择列表
      → Go 返回响应 → Vue 渲染 Markdown
  → 用户选择群 → 重复以上流程
  → 阶段推进: waiting_group → waiting_bids_filter → waiting_confirm
  → 用户确认 → HTTP POST → Go /api/v1/bids/dispatch
    → Go 查询待发送标讯
    → Go HTTP POST → Python /agent/dispatch/send
      → DingTalkSender.format_bid_message()
      → DingTalkSender.send_markdown()
      → 返回发送结果
    → Go 更新标讯 status=sent
```

## 6. 部署与容器化

系统使用 Docker 部署，Go 后端直接运行在宿主机，Python Agent 运行在 Docker 容器中。

### 6.1 服务地址

| 服务 | 地址 | 访问方式 |
|------|------|----------|
| Vue 前端 | `10.2.37.237:8088` | 浏览器直接访问 |
| Go API | `10.2.37.237:8088/api/v1` | 前端调用，需 Basic Auth |
| Python Agent | 内部 `:8000` | 仅 Go 后端调用 |

### 6.2 Python Agent Dockerfile

```
FROM mcr.microsoft.com/playwright:v1.60.0-jammy
WORKDIR /app
RUN pip install -r requirements.txt -i https://pypi.tuna.tsinghua.edu.cn/simple
RUN playwright install chromium
COPY . .
CMD ["uvicorn", "server.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

使用 Playwright 官方镜像作为基础，内置 Chromium 浏览器。

### 6.3 配置管理

配置文件路径通过环境变量 `CONFIG_PATH` 指定：
- 宿主机开发/测试：`configs/config.yaml`
- Docker 容器：`configs/config.docker.yaml`（`python_service_url` 指向容器名）
